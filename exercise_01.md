# Exercise 1

# 1️⃣ STM32CubeIDE Project Setup & First LED Blink

## What this demo should show
This exercise introduces the complete basic workflow for STM32 development:

- Create a new STM32 project using STM32CubeIDE
- Configure the board using CubeMX
- Generate initialization code
- Blink the onboard LED using HAL

This is the first hands-on exercise to help participants become comfortable with the STM32 development environment and basic GPIO control.


## What to configure first in CubeMX

* Board
- Select **STM32F446RE Nucleo**
* Peripheral / Pin configuration
- Enable onboard LED pin  
  - **PA5** → GPIO Output
* Clock configuration
- Keep default clock initially OR configure system clock as required
* Project generation
- Generate code using HAL

## Replace the while(1) loop with this

```c
while (1)
{
    HAL_GPIO_TogglePin(GPIOA, GPIO_PIN_5);
    HAL_Delay(500);
}
````

## Observation

* Onboard LED should blink every 500 ms
* Project build → flash → run flow becomes clear

📌 `This is the first successful hardware bring-up step.`

# 2️⃣ Understand Generated Project Structure

## Goal of this step

Before adding more code, one should understand the basic STM32 project structure generated by CubeMX / CubeIDE.

## What to observe in project structure

Typical files:

* `main.c`
* `gpio.c`
* `usart.c` *(if enabled later)*
* `stm32f4xx_hal_msp.c`
* `stm32f4xx_it.c`

## Key explanation

* `main.c` → application flow
* `MX_GPIO_Init()` → pin configuration
* HAL initializes device and peripherals
* User code must stay inside `USER CODE BEGIN / END` sections

📌 `Generated code gives initialization framework; application logic is added by the developer.`

# 3️⃣ LED Control using HAL API

## Goal of this step

Understand GPIO control using HAL abstraction.

## Replace the while(1) loop with this

```c
while (1)
{
    HAL_GPIO_WritePin(GPIOA, GPIO_PIN_5, GPIO_PIN_SET);
    HAL_Delay(500);

    HAL_GPIO_WritePin(GPIOA, GPIO_PIN_5, GPIO_PIN_RESET);
    HAL_Delay(500);
}
```

## Observation

* LED turns ON for 500 ms
* LED turns OFF for 500 ms

## What this demonstrates

This step shows explicit GPIO control using HAL.

📌 `HAL provides readable and portable APIs for peripheral access.`

----

# 4️⃣ LED Control using Direct Register Access

## Goal of this step

Compare HAL-based GPIO control with register-level control.

## Replace the while(1) loop with this

```c
while (1)
{
    GPIOA->ODR ^= (1 << 5);

    for (volatile uint32_t i = 0; i < 500000; i++)
    {
        // crude software delay
    }
}
```

## Observation

* LED blinks again
* No HAL GPIO API is used

## What this demonstrates

This step shows how GPIO can be controlled directly through registers.

📌 `Register access gives fine control and lower overhead, but reduces readability and portability.`

----

# 5️⃣ Compare HAL vs Register-Level Programming

## Goal of this step

Help participants understand when each style is useful.

## Summary

### HAL-based approach

* easier to read
* easier to maintain
* preferred for most application-level code

### Register-based approach

* hardware-specific
* more control
* useful for learning, optimization, and debugging

📌 `A good embedded engineer should understand both abstraction and hardware-level access.`

# 6️⃣ Add a Simple Helper Function for LED Toggle

## Goal of this step

Introduce small code modularization.

## Add this helper function above main()

```c
void led_toggle(void)
{
    HAL_GPIO_TogglePin(GPIOA, GPIO_PIN_5);
}
```

## Replace the while(1) loop with this

```c
while (1)
{
    led_toggle();
    HAL_Delay(500);
}
```

## Observation

* Same behavior as before
* Code becomes cleaner and easier to reuse

📌 `Even small embedded programs should be written in a modular style.`

***

# 7️⃣ Change Blink Rate

## Goal of this step

Show that behavior can be changed easily by modifying timing.

## Replace the while(1) loop with this

```c
while (1)
{
    led_toggle();
    HAL_Delay(200);

    led_toggle();
    HAL_Delay(800);
}
```

## Observation

* LED ON/OFF timing feels different
* Behavior is determined by application logic

## What this demonstrates

This introduces the idea that timing behavior is part of system design.

📌 `Embedded software is not only about logic, but also about timing.`

# 8️⃣ Fault Injection for Debugging Practice

## Goal of this step

Introduce the habit of debugging when hardware does not behave as expected.

## Intentional faulty code example

```c
while (1)
{
    HAL_GPIO_TogglePin(GPIOA, GPIO_PIN_6);
    HAL_Delay(500);
}
```

## Observation

* LED does not blink
* Program builds successfully, but hardware behavior is wrong

## What participants should do

* Check LED pin mapping
* Verify GPIO pin configuration
* Step through code in debugger
* Inspect the value and pin name carefully

📌 `Successful compilation does not guarantee correct hardware behavior.`

----

# 9️⃣ Debugging This First GPIO Application

## Goal of this step

Use basic debugger features to verify code execution.

## What to do

* Place breakpoint on `HAL_GPIO_TogglePin(...)`
* Run in debug mode
* Step over lines one by one
* Observe program flow

## What to observe

* Breakpoint hits repeatedly
* Code execution is reaching the LED toggle line
* If LED does not blink, issue may be configuration-related, not logic-related

📌 `Debugging starts by confirming whether the expected line of code is actually being executed.`
